# Contributing to Ideas Collection Project

Thank you for your interest in this project! We welcome all forms of contributions.

## Ways to Contribute

### 1. Submit Ideas

If you have a great idea, please follow these steps:

1. **Fork this repository**
2. **Create a new branch**: `git checkout -b idea/your-idea-name`
3. **Create your idea file in the `ideas/` directory**: Refer to `ideas/TEMPLATE.md`
4. **Commit your changes**: `git commit -m "[IDEA] Add your idea title"`
5. **Push the branch**: `git push origin idea/your-idea-name`
6. **Create a Pull Request**: Use the title format `[IDEA] Your Idea Title`

**PR description should include**:
- Brief overview of the idea
- Why this idea is valuable
- Any relevant background information

### 2. Vote for Ideas

- Browse open Pull Requests
- Use the üëç reaction on ideas you like
- Provide constructive feedback in comments

### 3. Implement Projects

If you want to implement a highly-voted idea:

1. **Express your intention** in the relevant Issue or PR
2. **Discuss the plan** with the idea author and maintainers
3. **Create a project repository** and start development
4. **Add project documentation** in the `projects/` directory:
   - `README.md` - Requirements description
   - `proposal.md` - Research report
   - `REPOSITORY.md` - Repository link
5. **Submit a PR** to update the project list

### 4. Improve Documentation

- Fix spelling or grammar errors
- Improve clarity of instructions
- Update outdated information

### 5. Report Issues

If you find a problem:

1. **Check if a related Issue already exists**
2. **Create a new Issue** and provide:
   - Clear title
   - Detailed description
   - Steps to reproduce (if applicable)
   - Expected vs actual behavior

## Code Standards

### Idea Files

- Use Markdown format
- Follow the structure in `TEMPLATE.md`
- Use lowercase letters and hyphens for file names

### Project Documentation

- Must include three core files: README.md, proposal.md, REPOSITORY.md
- Refer to `Example/` for format and content depth
- Keep documentation updated with code

### Git Commit Convention

Use clear commit messages:

```
[TYPE] Short description

Detailed description (if needed)
```

**Types**:
- `[IDEA]` - Add or modify ideas
- `[PROJECT]` - Add or update project documentation
- `[DOCS]` - Documentation improvements
- `[FIX]` - Bug fixes
- `[CHORE]` - Other maintenance work

**Examples**:
```
[IDEA] Add AI-powered code review assistant

[PROJECT] Add blockchain voting system documentation

[DOCS] Improve contributing guidelines

[FIX] Correct typo in README.md
```

## Pull Request Process

1. **Ensure your changes follow project standards**
2. **Update related documentation** (if needed)
3. **Clearly describe your changes** in the PR
4. **Wait for review and feedback**
5. **Make necessary modifications** based on feedback
6. **Once approved, the PR will be merged**

### PR Checklist

Before submitting a PR, please confirm:

- [ ] Followed file naming and structure standards
- [ ] Included necessary documentation
- [ ] Clear and concise commit messages
- [ ] No unrelated file changes included

## Code of Conduct

### Our Pledge

To create an open and welcoming environment, we pledge:

- **Respect** all participants, regardless of experience level, gender, sexual orientation, disability, appearance, race, or religion
- **Accept** different viewpoints and experiences
- **Gracefully accept** constructive criticism
- **Focus** on what is best for the community
- **Show empathy** towards other community members

### Unacceptable Behavior

- Using sexualized language or imagery
- Insulting/derogatory comments, personal or political attacks
- Public or private harassment
- Publishing others' private information without permission
- Other conduct considered inappropriate in a professional setting

### Enforcement

Participants who do not follow the Code of Conduct may be temporarily or permanently removed from the project.

## Voting Rules

### Idea Voting

- Anyone can vote on ideas in PRs
- Use GitHub's üëç (thumbs up) reaction
- One vote per GitHub account
- Vote manipulation or bot voting is not allowed

### Selection Cycle

- Monthly selection of the most voted ideas
- Selection time: Last day of each month
- Selected ideas will be tagged and assigned to volunteers

### Voting Transparency

- All votes are public (GitHub reactions)
- Maintainers will regularly publish voting statistics
- If vote manipulation is detected, the related idea will be disqualified

## Questions and Help

2. **Search existing Issues**
3. **Create a new Issue to ask**
4. **Discuss in the Discussions area**
